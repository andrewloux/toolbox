01

The seam: one ticket, two companies

born → filed → parked → woken

Since March, 15,622 issues have landed in the ATLASBUF Jira project; 15,529 are Physical Repair Tasks. The other 93: 49 General requests, 29 Deployment Tasks — an issue type, distinct from the Request Type value “Deployment” that repair tickets carry — 12 Access Requests, 3 legacy repair types. Most are auto-generated by Google's workflows — 93% of the corpus; the rest are filed by humans on both sides, Google hardware ops and FluidStack engineers alike.

The ticket is the pause/resume signal: the workflow files it and parks; FluidStack works it; the status transition is the wake-up. Google's own ticket footer says it plainly ↗: “Close this ticket after the issue is resolved to resume the workflow.”

Filing and managing a repair across the Google↔FS boundary has one confirmed interface today: the Jira ticket. Inside FluidStack, dcim-tasks is the system of record; Jira is the transport in front of it.

That seam is what the API replaces: POST /repairs in, status transition out.

The full boundary census — five cross-boundary touches; only one carries repair requests. Scope rule: the census counts signal channels — paths a repair request or work signal could ride between the companies. Standing access surfaces are deliberately not rows: Google's Teleport tenancy (the tenancy drill) and its direct NetBox/feldspar editor access (§04) are how Google's code and people are present on FS systems, not channels a request crosses — the same classification the Temporal map applies to Temporal Cloud.

01·0

Two deployments, one page

Every “today” receipt on this page comes from the first; the API is being built for the second:

Bloom · serving Atlas — buf101

Google's existing TPU system (Ironwood/Trillium pods) in FluidStack's New York site — also called CB3 (the Google-side name per a session-cited attribution — see the decoder).

The triangle

FluidStack operates the building and the floor · Google (Bloom) supplies and operates the TPU system and its repair automation (the operating split per FS — the tenancy drill hedges it, and no repo doc read states it) · Atlas — FS's codename for the customer, Anthropic (written down: “between Fluidstack (FS) and Anthropic (Atlas)”) — consumes the capacity. Atlas (org_17EFoZ7YaZi1mqh1), Google (org_siG1C0abE90FUnWo), and Dell (the hardware vendor — the Borneo card's actors row introduces it — org_j7mjrToZkciBXK1h, bastion-enabled) each hold scoped Auth0 access; one house external-org pattern covers all three, four orgs counting the BMS integrators (org_fXULDu3zQbqpamgA — the tenancy panel carries the terraform’s verbatim comment). BMS there is Ignition’s building-management system, not the board-controller BMC — a contrast no single doc states: this page assembles it from the terraform comment and the TPU hardware doc, and only the FISH RFC glossary defines both terms in one table. §04’s feldspar group list is a different, wider census — NetBox provisioning, not Teleport — keyed on the same Auth0 org mechanism (the org sets only partly overlap — google and atlas/anthropic are the shared ones): its “anthropic” is the terraform’s “atlas”, one org, two config labels.

The traffic

The ATLASBUF Jira project and the Ext-Bloom-Atlas-Workflows bot belong to this deployment; the project's 15,622 issues are its record — including the test phase: March's 17 are hand-filed simulations, and early tickets include wdl-staging tasks (Rob: “We are using WDL as a testbed and I have it configured to use ‘ATLASBUF’ for testing” ↗). 93% is filed by Google's bot, the rest by humans on both sides (the live queue).

Borneo — IAH1

Google's incoming B300 GPU cluster — Dell XE9780L machines (NVIDIA's GB300 platform; BP-232 uses both names) + Spectrum SN5600 switches — at IAH1, Stream DC's colo building, FluidStack's deployment to operate.

The actors

Neither Stream nor Dell appears in any repair-flow artifact this page read: Stream is the landlord; Dell, the hardware vendor, holds bastion-only support access (scoped in the sso terraform — the buf101-era FISH env; the pattern is org-level Auth0, and its IAH1 instantiation is unestablished like the rest of the stand-up). The triangle's third vertex is Google itself: Borneo is Google's own capacity, no Atlas-style third-party tenant — Google is both the repair requester and the consuming customer (Mike's TL;DR: “Borneo == BORg on NEOclouds, aka ‘Google running Vertex AI on us’; starting with 5k B300 in Stream Houston” (Vertex AI: Google's managed AI platform) ↗).

The dates

Contractual, per BP-232 (a ticket in BP, the Borneo-program Jira project): ready-for-service Jun 16 for data hall 2 (“the tightest constraint”), Aug 25 / Aug 31 for halls 1 / 3. Google asked for “machine-mgmt API” integration ↗ “8 weeks before hardware lands” (Mike: not reasonable ↗). “Machine mgmt” is the Partner-Ops doc's scope title; the FS-hosted surface inside that scope is what Google calls the “3P Ticketing System” and this page calls the Maintenance API.

The gap

No repair traffic yet; the API is specified for it. The rest of the gap — the fallback question — is the amber note below.

The fallback question — open, and burning

Borneo has no repair traffic yet, and the API being specified for it is deliberately generic so Atlas can later migrate onto it as a second tenant.

What Borneo uses if machines land before the API ships is undecided; the Jira seam is the only existing fallback — and “existing” means the buf101 instance: whether its fulfillment machinery (per-site dcim config, console queue, DCO hands, NetBox modeling) could stand at IAH1 in time is itself unestablished.

BP-232 tracks facility milestones, not hardware-arrival dates, so the fallback window's length is itself unknown.

So is which Jira project Borneo fallback tickets would even live in: ATLASBUF is the Atlas deployment's project, and TQ's (Thomas Toquothty, FS networking) ask to keep Borneo conversations out of Atlas's sight — failure mode 2 — argues against parking Borneo tickets there. Whether Atlas can actually see ATLASBUF isn't established.

01·MAP

The Temporal map — every engine on this page, in one place

Three engines appear in the configs read. Google’s runs in the google-system namespace on FISH — frontend temporal-frontend.google-system:7233 — and hosts the doctor workflow family (bloom-temporal on Google’s Gerrit and the temporal-web Teleport registration are the receipts).

Temporal Cloud is the vendor’s hosted service — Temporal Technologies’ SaaS — and infrastructure.gdjny is FS’s own namespace on it (the config lives in FS’s systems repo), reachable at ap-northeast-1.aws.api.temporal.io:7233. It is live today as site-worker’s primary connection; being FS’s estate, it is not a Google↔FS boundary touch. What that primary carries today wasn’t read, and “gdjny” is a bare token — its expansion isn’t written down. FS’s own dcim engine runs dcim-worker’s RepairTaskWorkflow — its production home wasn’t read either; the example config defaults to localhost:7233 — if that engine also rides Temporal Cloud in prod, the census collapses to two.

Who connects to what: site-worker (the per-site worker service in the systems repo) holds both connections in one base config — values.yaml names Temporal Cloud as the worker home and temporal-frontend.google-system:7233 as a localTemporalInstance, dialed as a client only.

Corrected Jun 11 — the direction was wrong here. worker.New() appears exactly once in site-worker, against the Temporal Cloud client (main.go:87), so its activities — machine netboot/reset, NetBox reads, ping checks (the activities tree) — register only there, and a workflow on Google’s engine cannot call them: Temporal can’t schedule activities across clusters. The earlier “on the secondary it executes the FS-side activities Google’s workflows call” reading is retired; if rack-turnup’s QA phase resets machines, it does that through Google’s own workers, not site-worker. The google-system dial serves the opposite direction: RunSiteLocalWorkflow starts a named workflow on the local instance and polls it to completion, and its doc comment sets the direction outright — “Callers on the central Temporal instance invoke this as a child workflow, targeting the task queue of the desired site-worker, which then drives the local instance interaction” (workflow.go:36-39).

So site-worker is FS’s arm at the site — netboot/reset/ping need device-network reachability, and only something in-cluster can reach temporal-frontend.google-system — and FS’s gateway into Google’s Temporal, not their activity worker. Its queue google-workflows-site-worker-<site> is a Temporal Cloud queue (buf101 values:3) — the prefix describes purpose, not location. Zero callers of RunSiteLocalWorkflow (or of site-worker’s own ProvisionRackWorkflow/ProvisionMachineWorkflow) exist outside site-worker in either repo: the whole FS→Google start path is built but unexercised.

dcim-api’s google.go names Google’s RackTurnupWorkflow on queue rack-turnup-task-queue — and the trigger is gated off in code: TURNUP_WORKFLOW_DEPLOYED = false, “Google has yet to actually deploy this workflow” (rack_workflows.go). Production config answers the endpoint question: dcim-api dials Temporal Cloud — ap-northeast-1.aws.api.temporal.io:7233, namespace infrastructure.gdjny (argocd values) — not the google-system frontend. (An earlier draft inferred a same-frontend link from a queue name; retired as wrong.) Nor does site-worker host rack-turnup: it registers exactly three workflows — ProvisionRack, ProvisionMachine, RunSiteLocal ↗ (the Jun 10 tree; an April read showed a fourth, Firmware, since removed); the executing worker would be Google-deployed on that queue, which doesn’t exist yet — the code’s own TODO wonders whether their deployment will suffix it. The same file’s later TODO asks whether the dial “should be a separate temporal client for the on-prem deployed control plane?” (rack_workflows.go:159) — the bridge reads as that answer: the sanctioned FS→Google start would be a RunSiteLocalWorkflow child on site-worker’s queue, not a direct dial. Nothing calls it yet.

The kickoff set the API build’s direction as “Temporal Cloud + site-local workers.” Temporal Cloud is already live as site-worker’s primary, so the direction names an estate that exists. Open: whether the API build rides that estate or adds its own, and whether “site-local workers” means the existing site-worker service or new workers.

Union, not mirror

FluidStack runs its own problem-code namespace — the hospital/healthcheck flow: FS's intake for sick machines outside Google's doctor loop (the hospital-rack process pulls failed hardware; healthcheck qualification returns it). No counterpart in Google's proto.

So the API's registry isn't a copy of repairs.textproto: it hosts two vendors' vocabularies and serves unknowns as warnings, not rejections.

The “structured” channel, unvalidated

customfield_12271 is the field dcim-tasks' config reads as its Task ID, populated on 99 of the 1,112 human tickets. In the wild it carries rack locators (r0113a), whole prose sentences, display-name forms (“Power Cycle”), and test debris ([smoke-test-device-01], [buf101-e2e-claude-test]).

The sync code explains the name/role split: dcim-tasks reads this field into the task row's problem_codes column — reconcile.go calls it the “problem codes field” ↗ while the config key says Task ID, historical drift between key and role. cf_12271 is the de-facto code carrier, which is exactly why its anarchy matters.

Jira accepts all of it silently — nothing validates this field today. The API's create path has to.

What the grammar can say

Twelve entity kinds; the finest TPU granularity is a tray (machine + index), with free-string sub_entity_ids below that. A link is a pair of port references.

Fired in production

Five of the oneof's types in the machine-filed corpus — link 3,416 · machine 1,692 · port 186 · tpu_tray 66 · host_tray 56 — plus one stray label outside it (fru ×1 — field-replaceable unit). The human custom-field pass adds a sixth oneof type, switch (50) — same merge rule as the +3 fired codes. The oneof isn't aspirational; it's the schema the corpus already speaks.

Missing: cable · rack · DCN

The June 3 field test asked hands to reseat a cable the grammar can't point at. Human custom fields — the entity tokens ride the Device Name field, cf_11171 in the field map — add switch: (50, an in-oneof type, counted above) plus rack: (12) and 352 bare untyped names — both outside the grammar.

The grammar needs a rack type and a migration story for untyped legacy IDs — and DCN stays structurally unaddressable: zero typed coverage across the 1,079 free-text-bucket tickets where the cleaning campaigns live.

And the whole grammar is TPU-derived. What B300 repairs will reference is unmodeled: GPU tray? NVLink switch, link, domain? (NVLink = NVIDIA's GPU-interconnect fabric — the B300 era's analogue of ICI.) CUJ1 — CUJ = Critical User Journey, the Partner-Ops doc's unit (§03) — literally names “Machine and NVLink switch intake”. Extending EntityId for GB300 hardware is a spec input of the same rank as the code namespace.

How we respond to them, today

The Jira status transition is the only confirmed reply channel. The machine-truth lives in TPU-CC: GetProblem / ClearProblem + PerformHealthTest are the problem lifecycle on their side, and nearly every auto-filed ticket cross-links its TPU-CC diagnostics URL. The 1,051-of-14,510 figure = machine-filed bodies containing neither a WorkflowID: line nor a Temporal run URL — no workflow string at all; it's one count, reused wherever 1,051 appears, and TPU-CC-link coverage rides the same template, not separately counted.

Whether FluidStack ever calls those RPCs today — for instance from the QA-handoff button — is unverified from our side.

How we map them

The repair-tuple ticket is the Problem message + its registry entry, serialized to prose: 【Problem Code】= problem_code · 【Target Entity】= EntityId · 【Diagnostic Data】= description.

POST /repairs un-serializes it — same fields, typed again (whether 【Requested Actions】 rides along is the open boundary question in 01·C's anatomy row).
