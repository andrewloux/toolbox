04

Foundry: name the destination, read the plumbing

where it mounts · what it reads

FluidStack's infrastructure truth lives in NetBox (the TPU sites' instance is named feldspar ↗), which everyone — Systems, Atlas, Google, DCIM — reads directly, and which Jason Legler describes as “so tired” — “NetBox REALLY wants us to replace it” ↗.

The consumer list is a working-session citation[10], but the access plumbing is now read from config: dcim-api-feldspar provisions NetBox users into feldspar-editor/-operations/-viewer groups keyed by Auth0 org — google (editor), google-contractor and ies-contractors (operations; “IES” is an external-contractor Auth0 org — its expansion isn't written in anything read), google-viewonly and anthropic (viewer) ↗.

Multiple stores synced expensively produce staleness bugs — the SPOCS rack-role drift between wdl101 and buf101 was exactly this class — Sean Banko's receipt, quoted in full below ↗.

Foundry is the replacement: “the platform containing the data center model, hardware state” (Scott McCoy), an aggregator of tier-2 services — the per-domain systems in the table below (DC Fab, Tally, Crucible, BlueBook), with Foundry as the single face over them[7]. Platform RFC: foundry PR #1233 → projects/foundry-api/rfcs/00-foundry-platform.md.

Deployment status unestablished: the foundry repo holds no deploy receipts beyond the RFC. Whether a live instance exists is a Wk-2 shape-check question for Jason + Tyler.

The consolidation plan one level up is written though: Anshul's Platform Consolidation page sketches a stateless API Proxy Layer (Kaiser API, Partner API, DCIM API) with “Partner API introduced” as its roadmap step 4 ↗ — and step 1 landed Jun 10: dcim-console moved into the Kaiser repo (#2525); partner-api and partner-console already exist as services there. Name care — a third collision: that FS-side “Partner API” is a Go proxy between the partner console and Kaiser (its own docs say exactly that) — not Google's Partner Operations API (03), and not this Maintenance API; whether foundry-gateway and that proxy layer converge isn't designed anywhere read.

“We are trying to kill crucible tally and dcfab from yall perspective in general and just call it foundry. you shouldnt have to care where it comes from.”

Anshul — Jun 2, #proj_borneo ↗ message[8]

The dual: Foundry is mid-migration. NetBox remains the operational source of truth; Foundry components are fed by dual-writes and periodic syncs; “not prod ready yet Rob” was Anshul Kamath’s answer when Rob came asking — the April CMDB thread ↗.

Sean's posture is the dependent's reality: “Foundry / DC Fab gets to decide what right is, I'm just a dependent of that decision.”[10] (“Sean” is Sean Banko — the thread is recovered ↗; the line continues: “I assumed/expected the role name in wdl101 to be consistent with buf101, and that turned out not to be true, which is what broke me.”)

Spec consequence: name the API's entity references against Foundry's resource model (Jason + Tyler are the shape-check) — Foundry supplies the names, Google's EntityId supplies the type grammar (01·D) — knowing the implementation reads NetBox-backed data for the foreseeable future. Design the contract for the destination, not the current plumbing. For IAH1 specifically, which NetBox models it is unestablished: the linked site registry carries only two entries ↗ — Appleton, the registry's name for the buf101 site (“Appleton, Buffalo”, slug buf1), and Woodlands/wdl1 — no IAH1.
