One doc, two API surfaces, opposite hosts

the doc demanding this API

Google is putting a B300 cluster (Borneo — “BORg on NEOclouds”, per Mike ↗) into IAH1 — a Stream DC colo facility where FluidStack operates the deployment. The contract for who touches a broken machine, who tracks every serial, and who gets told when something changes is the Google-authored Borneo Partner Operations API doc[1].

The terminology trap Thomas Barrett untangled on June 8: the doc describes two surfaces, hosted by opposite parties[2]. One naming overlap to hold onto: the doc's title — Partner Operations API — doubles as the name of its Google-hosted surface below; the FS-hosted one is the Maintenance API.

Two return channels, different payloads: the Maintenance API's webhook carries repair lifecycle status (the CUJ3 interaction); the Borneo adapter's completion sync — the CUJ2-shared “on change in Crucible” mechanism — carries asset/inventory updates to Google's asset database after the physical work. They answer different questions and don't replace each other.

Failure mode 1 — the doc is necessary, not sufficient

Requirements exist that are only verbal. BP-290's (BP = the Borneo-program Jira project) DHCP-option toggle (“Fluidstack QA image vs Google BorneoOS — flip a config bit on the switch”) — BorneoOS reads as Google's machine OS image for the program, by contrast with the FS QA image; the name appears only in this requirement — made Sean go looking: “I don't actually see any information about this in the Borneo Underlay Routing Design document” ↗ (the Underlay Routing Design doc is a separate Google-authored network design document for Borneo — not the Partner Ops doc, and not read for this page). It falls out of a meeting from months ago.

Input gathering has to mine the verbal layer (Rob, Thomas Barrett, Mike), not just the doc.[5]

Failure mode 2 — two Google teams, colliding names

CUJ numbers don't transfer between Google teams. The Borneo program and the Bloom (TPU) program are run by different Google counterparts, so a CUJ numbered by one says nothing about the other's list — the two deployments' Google-side casts on this page don't overlap (FS people span both, Rob foremost; Google's internal org structure isn't established). The numbering also collides with demo CUJs for Kaiser, an unrelated FluidStack product.

TQ — FS networking[10] — asked for separation: “We do need to keep the Borneo convos a bit out of sight from Atlas though so we should be mindful of overlap there” — Thomas “TQ” Toquothty, May 26 ↗. Atlas is the existing tenant; the confidentiality runs customer-ward.

Naming discipline in the spec (“Borneo CUJ3”) prevents boundary blur.[6]
