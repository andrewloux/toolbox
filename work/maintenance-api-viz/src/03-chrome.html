The Maintenance API:
how repairs flow today, and the contract replacing the Jira seam

Google is landing a B300 GPU cluster — Borneo — at IAH1, a Stream DC colo facility where FluidStack operates the deployment. Its repairs are slated to flow through the API this page motivates and constrains; the spec is planned for ~Jul 21. The repair machinery that exists today serves Google's earlier TPU deployment (Bloom) at buf101, run for the existing tenant (Atlas): Google's workflows decide what gets repaired, FluidStack does the how, and the repair request itself travels only as a Jira ticket. That machinery is the precedent the API generalizes.

FS's working plan has Borneo never touching the Jira seam[10]; Google's acceptance of that timeline is unestablished. If machines land before the API ships, the fallback is undecided — the Jira seam exists at buf101 only, and the fallback note in 01 carries the question whole. Atlas leaves Jira only if Rob's pitch to move its traffic onto the same API lands — a workstream pursued in parallel, not a blocker.

This page walks that machinery end-to-end, then the API contract replacing it. Every solid box drills down (dashed boxes are pointers); nearly every claim links a receipt — the few whose permalinks weren't recovered are named, individually, in [10].
