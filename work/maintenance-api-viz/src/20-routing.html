02

Interrupt or work item? One form field carries the signal

storms queue · pages stay rare

One form field decides whether a repair becomes a page or a queue entry: the JSM (Jira Service Management) Request Type — uniform on the 200/200 live pull (an in-session Jira pull; the archived corpus crawl is a different dataset, and the crawl's field set lacked request type). Whether the field is uniform across every issue type wasn't checked. Behaviorally the discriminator holds: June 1–3, “Deployment” alerts created zero incidents and paged no one; all 11 incident-creating alerts carried non-Deployment queues. The route expressions themselves are dashboard-only — the verified note below carries the Jun 10 re-check, its limits (other attributes can't be fully excluded), and the receipts.

The two intake watchers are independent listeners, not a fork; a third watcher — Google's parked workflow — sits on the status field, not intake. June 2's mis-typed ticket (the red note below) carried a queue — “Compute - Cable Repair”, request type 1414 (a request type, strictly; incident.io's attribute names these “queues” and the page follows its usage) — with no entry in dcim-tasks’ production RequestTypes map — only Deployment (1669) and Compute or Network Device Repair (1415) are mapped ↗ — so the mirror had no mapping to sync it under.

Why the split exists: repairs arrive in storms. The earlier inference here (“repairs never reach incident.io”) was wrong in mechanism, right in outcome. The “Jira queues CB3” alert source (id 01KKW2SAETESKT6K806GNHJCM7) ingests ATLASBUF broadly: machine-filed repairs arrive as alerts carrying the ticket's Request Type, reporter, and full body as attributes — and always at priority P4; the source never sets the org's “Should Page” attribute.

What keeps pagers quiet is the layer above: the source's 50 newest alerts are all repair tickets — all P4, zero attached to incidents. Paging happens only when an alert's attributes match an incident-creating route, and that routing config is dashboard-managed (the house terraform, cluster-ops recipes, holds only the alertmanager (Prometheus's alert router) source). The route expressions themselves remain dashboard-only — re-confirmed Jun 10 against the live API, which exposes the route’s existence (one alert route, owned by the Systems Engineering team) but not its conditions. What it does expose is two-stage behavior: a matching alert raises a triage incident carrying a Team field, then the “DC ops Alerting path” workflow fires the escalation to that team’s path (Systems Esc, 5-minute ack). The June 1–3 behavioral receipt, observed through the same API: 11 alerts created incidents, every one carrying a non-Deployment queue (urgent-request tests, plus June 2’s misroute); “Deployment” alerts in the window created none.

Also code-confirmed: the bridge is synthetically monitored (jira-heartbeater files an “Internal Alert” each cycle, expects a firing alert within 120 s; whether it exercises the same “Jira queues CB3” source repair tickets traverse is unread — the Path A panel’s caveat), and alerts auto-resolve when the ticket closes — 13 of the 50 sampled already had. 02·B

Inside Path B — ticket → hands → Done

Before the pipeline, the three things it threads together — the page has been using these names without introducing them:

actorrole in the loop
DCOData-Center Operations — the humans at the site. Technicians (“hands”) walk the floor, do the repair, and mostly route themselves (self-claim); a dedicated lead-assignment role was formalized Jun 9 (01·LOOP ⑥).
dcim-tasksThe work-order system. Every repair becomes a task row — status, assignee, pinned SOP version, console deep-link. It's the system of record for floor work; Jira is just the transport in front of it.
SOPThe reviewed, versioned procedure resolved from the problem code (full lifecycle: 02·C). Rendered as guide panels on the task — tells the tech what to do and what evidence to capture.

The loop between them: a ticket becomes a task; the tech claims it (self-assign is the observed norm — 01·LOOP ⑥; a lead can route); the tech opens it in the console and follows the SOP's panels. On completion, dcim-tasks' outbound sync writes the status transition to the Jira ticket — the outbound half of the jira.go status map. That Done transition is what wakes Google.

That's the hands path. The sweep flood closes differently: 99% of Jun 3's Dones were never assigned, closed by automation in a median 34 minutes (whose automation isn't determinable from the crawled fields); gaps the tech finds in the procedure loop back into the SOP library (02·C). How it fits the whole: a repair request carries a problem code (when one rides cf_12271 — the pipeline's stage-3 PIN panel (02·B) scopes how often that is); the code resolves to an approved, versioned procedure; the version freezes onto the task before hands move.
