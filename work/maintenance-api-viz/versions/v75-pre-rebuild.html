The Maintenance API:
how repairs flow today, and the contract replacing the Jira seam

Google is landing a B300 GPU cluster — Borneo — at IAH1, a Stream DC colo facility where FluidStack operates the deployment. Its repairs will flow through the API this page motivates and constrains (the spec itself lands ~Jul 21). The repair machinery that exists today serves Google's earlier TPU deployment (Bloom) at buf101, run for the existing tenant (Atlas): Google's workflows decide what gets repaired, FluidStack does the how, and the repair request itself travels only as a Jira ticket. That machinery is the precedent the API generalizes — FS's working plan has Borneo never touching the Jira seam[10] (Google's acceptance of the timeline is a separate, unestablished matter), though the Jira seam is the only existing fallback if machines land before the API ships; Atlas leaves Jira only if Rob's pitch to move its traffic onto the same API lands — a workstream pursued in parallel with the Borneo spec, not a blocker for it.

This page walks that machinery end-to-end, then the API contract replacing it. Every solid box drills down (ghost boxes are pointers); nearly every claim links a receipt — the few whose permalinks weren't recovered are named, individually, in [10].

01

The seam: one ticket, two companies

born → filed → parked → woken

Since March, 15,622 issues have landed in the ATLASBUF Jira project — 15,529 of them Physical Repair Tasks (the other 93: 49 General requests, 29 Deployment Tasks — an issue type, distinct from the Request Type value “Deployment” that repair tickets carry — 12 Access Requests, 3 legacy repair types) — most auto-generated by Google's workflows (93% of the corpus), the rest filed by humans on both sides: Google hardware ops and FluidStack engineers alike. The ticket is the pause/resume signal: the workflow files it and parks; FluidStack works it; the status transition is the wake-up. Google's own ticket footer says it plainly: “Close this ticket after the issue is resolved to resume the workflow.”

Filing and managing a repair across the Google↔FS boundary has one confirmed interface today: the Jira ticket (inside FluidStack, dcim-tasks is the system of record — Jira is the transport in front of it; the QA-handoff button at closure — a console checklist control hands fire on finish, detailed in 02·B — is a second, narrower cross-boundary touch whose Google side is unverified). That seam is what the API replaces: POST /repairs in, status transition out.

The full boundary census: five touches. The Jira ticket alone carries repair requests; the QA-handoff button touches the repair loop at closure (shipped on the OCS/TPU rack checklists — whether ordinary repair-task closure fires it, or only rack-checklist work, isn't established); JSM's transition emails ride the Jira seam itself, counted inside touch one. Three more touches aren't repair paths at all. google.go starts rack-turnup — deployment turn-up, not repair (though its QA failures are what file most repair tickets — 8,737 of the corpus' 15,529 repair tasks reference its runs — all 8,737 are repair tasks, verified, so the same count sums into the 15,622-wide decomposition in 01·A's family panel). TPU-CC's RPCs read and clear problem state and run health tests — diagnostics, not a way to ask FluidStack to fix something. And site-worker's secondary connection (a localTemporalInstance; its primary is Temporal Cloud — the u-fs panel splits the two) registers it on Google's Temporal to execute FS-side activities for their workflows (machine netboot/reset, NetBox reads, ping checks — the registration and activity code are the receipts; live dispatch traffic wasn't read) — execution plumbing inside the workflows, not a way to request a repair.

Two deployments, one page. Every “today” receipt on this page comes from the first; the API is being built for the second:

Always-on, both paths: JSM (Jira Service Management) emails the reporter + request participants on every transition — for machine-filed tickets, that reporter is the bot.

The diagram draws the common case as a fork; strictly, the two paths are independent listeners on the same tickets — Request Type is the observed key; 02 refines the route mechanics — June 2's mis-typed ticket paged the on-call while being queue-class work by nature — a cable repair for hands (whether the mirror also synced it despite the wrong Request Type is hidden with the rest of the unread config) (02).

FluidStack holds no trigger for the repair doctors proper (machine-doctor, machine-doctor-ocs-link, link-repair) — no FS code path starts them. rack-turnup — drawn in the same family box — is the one exception: dcim-api/…/google.go starts RackTurnupWorkflow by string name, and that workflow's QA failures are what file repair tickets.

What actually changes: the transport. The decide-WHAT / do-HOW split stays; dcim-tasks stays the fulfillment backend. One channel doesn't survive the move: JSM's always-on transition emails exist only because Jira is the transport — for API-borne traffic their only successor is whatever d-push decides.

One thing this picture doesn't redraw: Path A. API-borne requests never touch the Jira watcher incident.io keys on, so what pages a human for a page-shaped request has to be re-wired against the routing-class field — build-phase design, not yet decided.

The ticket becomes a resource with explicit lifecycle — Rejected is already a real Jira terminal state; 2,671 repair tasks (plus 15 non-repair rejections; 17% of the corpus) ended there with no contracted signal back to the caller — JSM's transition email fires, but nothing contracts it. The status watch becomes contractual — this page argues webhooks; webhook-vs-poll is open decision d-push. Storm-shaped load gets batch + coalescing as first-class verbs.

evidenceStructured payload (measurements + thresholds), not free text — the June 4 campaign's tickets arrived schema-less and 76% were rejected (87 of 114; the templated Jun 3 sweep closed 99% Done, and of those Dones 99% closed unassigned, median 34 minutes — two near-twin ratios by coincidence: share-Done of the sweep, share-unassigned of the Dones. Union, not mirror

FluidStack runs its own problem-code namespace — the hospital/healthcheck flow: FS's intake for sick machines outside Google's doctor loop (the hospital-rack process pulls failed hardware; healthcheck qualification returns it). No counterpart in Google's proto.

So the API's registry isn't a copy of repairs.textproto: it hosts two vendors' vocabularies and serves unknowns as warnings, not rejections.

The “structured” channel, unvalidated

customfield_12271 — the field dcim-tasks' config reads as its Task ID, populated on 99 of the 1,112 human tickets — carries in the wild: rack locators (r0113a), whole prose sentences, display-name forms (“Power Cycle”), test debris ([smoke-test-device-01], [buf101-e2e-claude-test]). The name/role split is now explained by the sync code: dcim-tasks reads this field into the task row's problem_codes column (reconcile.go calls it the “problem codes field” while the config key says Task ID — historical drift between key and role). cf_12271 is the de-facto code carrier, which is exactly why its anarchy matters.

Jira accepts all of it silently — nothing validates this field today. The API's create path has to.

The correlation key already exists

cf_14593 is a UUID on 657 of 1,112 human tickets, mapped in dcim-tasks' Jira schema — the API adopts it as the create-side idempotency/dedup key rather than inventing one.

What the grammar can say

Twelve entity kinds; the finest TPU granularity is a tray (machine + index), with free-string sub_entity_ids below that. A link is a pair of port references.

Fired in production

Five of the oneof's types in the machine-filed corpus — link 3,416 · machine 1,692 · port 186 · tpu_tray 66 · host_tray 56 — plus one stray label outside it (fru ×1). The human custom-field pass adds a sixth oneof type, switch (50) — same merge rule as the +3 fired codes. The oneof isn't aspirational; it's the schema the corpus already speaks.

Missing: cable · rack · DCN

The June 3 field test asked hands to reseat a cable the grammar can't point at; human custom fields — the entity tokens ride the Device Name field, cf_11171 in the field map — add switch: (50 — an in-oneof type, counted above) plus rack: (12) and 352 bare untyped names, both outside the grammar.

The grammar needs a rack type and a migration story for untyped legacy IDs — and DCN stays structurally unaddressable: zero typed coverage across the 1,079 free-text tickets where the cleaning campaigns live.

And the whole grammar is TPU-derived. What B300 repairs will reference — GPU tray? NVLink (NVIDIA's GPU-interconnect fabric — the B300 era's analogue of ICI) switch, link, domain? CUJ1 (CUJ = Critical User Journey, the Partner-Ops doc's unit — §03) literally names “Machine and NVLink switch intake” — is unmodeled. Extending EntityId for GB300 hardware is a spec input of the same rank as the code namespace.

Inside Path B — ticket → hands → Done

Before the pipeline, the three things it threads together — the page has been using these names without introducing them:

actorrole in the loop
DCOData-Center Operations — the humans at the site. Leads triage and route the queue; technicians (“hands”) walk the floor and do the repair.
dcim-tasksThe work-order system. Every repair becomes a task row — status, assignee, pinned SOP version, console deep-link. It's the system of record for floor work; Jira is just the transport in front of it.
